### 1.3 技术管线（全部复用 A 的成熟做法）

- 抠图：rembg（isnet），B 的纯色蓝灰背景比 A 的米色底更好抠。
- 眼神跟随：连通域程序实测眼位（B 的青蓝玻璃眼色块分离度高）→ OpenCV 抹眼制无眼底图 → CSS 重建可动眼。**风险注记：写实玻璃质感的高光比卡通复杂，CSS 眼的以假乱真程度需实测一轮，A 踩过的坑（手工测坐标）不会再犯。**
- 尾巴/翅膀状态动效：MVP 用 CSS 对图层做轻微摆动/发光，不做骨骼动画（遵守原文档 §7.2 暂不做清单）。

## 3. 演示动线（与 A 完全对齐，公平对比）

三幕结构与 A 相同，评审时相同操作走两遍：

1. **第一幕 · 桌宠是活的**：Lumo 趴在屏幕右下 → 呼吸/眨眼/眼神跟随 → 悬停照顾菜单（B 语境改为：🪰 喂虫子？或保持 🍡 通用零食 → 待定）→ 主动天气播报。
2. **第二幕 · 点它干活**：单击唤面板（Lumo 从面板边缘探头）→ 工具调用四阶段（整理线索 → 看图 → 翻文件 → 完成跳跃）→ 三通道联动（面板卡片 + 桌宠姿态 + 菜单栏图标）→ 出错态。
3. **第三幕 · 组件总览**：B 版状态姿态墙 / 菜单栏六状态 / App Icon 方向 / 冷调色板 Token。

**评审对比维度**（走完两遍后打分）：

| 维度 | 看什么 |
|---|---|
| 第一眼气质 | 治愈陪伴感 vs 灵动助手感，哪个更想长期放桌面 |
| 状态表达力 | 尾巴/翅膀（B 天然状态天线）vs 姿态+道具（A） |
| 品牌叙事 | 天一阁文化转译（A 有独家故事）vs 通用科技萌宠（B 无文化包袱但也无护城河） |

| 延展潜力 | 表情包/周边/官网哪个更能打 |

## 4. 待困困拍板的决策点（执行前必答）

1. **命名**：建议定 **Lumo**（原文档亦首推）；备选里的 *Mimo* 强烈建议放弃——与看图模型 MiMo 撞名，产品内外都会混淆。
2. **演示深度**：快速版（6 现成姿态，2–3 小时）还是完整版（+生图补 3 姿态，多半天、有一致性风险）？建议先快速版，对比胜出再补全。
3. **面板换肤**：按本方案做 B 专属冷调皮肤（推荐，对比才有意义），还是先共用 A 皮肤只换角色（更快但失真）？
4. **照顾互动的 B 语境**：喂食物件（虫子？光点？）——写实生物吃「小团子」会出戏。

## 5. 风险与如实提醒

- **写实风格的 CSS 眼**：玻璃眼球高光复杂，重建效果可能略逊卡通版，需一轮实测——最坏情况降级为「整头微转跟随」（不重建眼球，无双层眼风险）。
- **生图补姿态的一致性**：AI 补的 3 张与原设定图的鳞片/翼脉细节可能有偏差，快速版不受此影响。
- **道具已烘焙进姿态图**：看图卡、文件夹等发光道具在原画里，演示够用；正式产品若要道具与角色分离（道具跟随真实内容变化），需重新出无道具姿态。
- **两套 IP 并行的维护成本**：评审后建议尽快二选一，避免双 IP 资产长期并行。
